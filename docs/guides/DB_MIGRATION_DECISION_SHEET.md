# DB 마이그레이션 비교 체크 시트

> 스크럼 설명과 기준 문서를 비교해 작업 기준과 문서 수정 필요 사항을 빠르게 체크하는 문서

**최종 업데이트**: 2026-04-09  
**작성일**: 2026-04-09

---

## 사용 방법

- 각 항목별로 스크럼 설명과 기준 문서를 비교한다.
- 실제 작업은 `docs/erd`, `docs/API.md` 기준으로 진행 여부를 기록한다.
- 기준 문서에 누락이나 오류가 있으면 수정 제안 내용을 함께 남긴다.
- 이 문서는 `SCRUM-112` ~ `SCRUM-121` 착수 전 비교 체크 시트로 사용한다.

## 항목별 메모

### 1. 멀티테넌시 FK 기준

- 현재 구현은 이미 `company_code` 기준
- 로그인 로직과 JPA 매핑이 이 전제를 사용 중

### 2~4. documents 테이블

기준 문서:

- `docs/erd`
- `docs/API.md`

비교 참고 구조:

```text
documents
- id
- company_code
- title
- content
- document_type
- department
- form_file_url
- is_active
- created_at
- updated_at
```

진행 메모:

- ERD/API 기준상 문서 본문 컬럼이 필요
- AI 검색/답변 연동이 본문 기반 구조를 전제함
- 스크럼은 현재 구현의 `file_path` 기준이 아니라 기준 문서를 따라 정렬해야 함

### 5. chat_messages.company_code

- 스크럼 설명에는 검토 여지가 남아 있을 수 있음
- ERD 기준 `chat_messages`에는 `company_code`가 없음
- 이번 작업은 기준 문서를 따라 `user_id` 중심으로 진행

### 6. 상태값 컬럼 타입

- 기준 문서에는 표준값이 정의되어 있고, DB 타입 정책은 별도 판단이 필요함
- 현재 작업은 `VARCHAR` 우선 검토로 두고, 필요 시 타입 정책을 문서화

### 7. user_activity_logs.event_target

- ERD 기준 `event_target`이 존재함
- 스크럼 설명이 축약되어 있더라도 이번 작업은 유지 기준으로 진행

### 8. day_offset

API 기준:

- D+0 = 1
- D+3 = 4
- D+7 = 8
- D+30 = 31

### 9. 마이그레이션 도구

- 현재 구현은 `schema.sql` 자동 실행 방식
- 실제 전환 여부는 실행 절차와 협업 방식까지 같이 보고 정리

### 10. docker 검증 범위

- 우선 MySQL + Backend 기준으로 실행 가능한 검증 절차를 정리
- AI 포함 여부는 이번 스크럼 필수 범위와 분리해서 본다
