## AI 온보딩 에이전트: 위드버디(WithBuddy)

"낯선 첫 출근 눈치 보던 질문이 확신이 되는 순간, 'AI 온보딩 동료, 위드버디'"

## 2. 배경 및 컨텍스트 (Background)

### **온보딩의 중요성은 이미 알려져 있다 — 문제는 실행 여력의 격차다**

자원이 충분한 조직에서는 온보딩이 조직 적응과 유지율 관리를 위한 기본 장치로 운영된다. 고용노동부·한국고용정보원(2023)이 매출액 상위 500대 기업 인사담당자 315개사를 대상으로 조사한 결과, 온보딩 미운영 비율은 3.8%에 그쳤고 응답 기업의 79.4%는 온보딩이 조기퇴사 방지에 도움이 된다고 평가했다.
반면 같은 조사에서 응답자들이 꼽은 일반기업의 온보딩 미활성화 이유 1위는 조직·예산 부족(54.6%)이었다. 중소기업일수록 전담 HR이나 온보딩 전용 리소스를 갖추기 어렵고, 그 공백은 자연스럽게 사수와 동료에게 전가된다.

---

### 💬중소기업 신입사원에게는 눈치 보지 않고 물어볼 수 있는 창구가 없다

입사 첫날, 신입사원이 가장 먼저 느끼는 감정은 설렘이 아니라 눈치다. 연차는 언제부터 쓸 수 있는지, 비품은 어디서 신청하는지, 이메일/VPN은 어떻게 설정하는지 등 사소해 보이지만 당장 알아야 하는 것들 앞에서 신입은 멈춘다. 사수가 바빠 보이고, 아직 파악을 못했냐는 말이 돌아올까봐 질문을 삼킨다.
자체 설문조사(n=90)에서 신입사원이 입사 초기 가장 큰 어려움으로 꼽은 1위는 **'눈치 보여서 묻기 어려움(41.1%)'**이었다. 혼자 해결하려다 시간을 낭비하거나(27.8%), 더 큰 실수로 이어지는 악순환이 반복된다.
이 문제는 개인의 소심함이 아니라 **구조적 공백**이다. 그리고 그 공백은 단순히 문서가 없어서 생기는 것도 아니다. 심층 인터뷰에서 "SOP가 있어도 어떤 내용을 찾기 위해 무슨 SOP를 봐야 하는지 몰라 사수에게 묻게 된다"는 진술이 나왔다. 문서가 있어도 탐색 경험이 없으면 결국 사람에게 묻게 된다. 위드버디는 바로 이 지점, 즉 **문서를 질문 가능한 창구로 바꾸는 방식에 주목한다.**

**위드버디가 해결하려는 것은 초기 적응 실패를 증폭시키는 병목에 집중한다**

조기퇴사의 원인은 복합적이다. 직무 적합성 불일치, 조직 문화, 근로 조건, 보상, 커리어 기대 차이 등 여러 요인이 얽혀 있고, 그 모든 것을 기술로 해결할 수는 없다. 위드버디는 그중에서도 **초기 적응 실패를 증폭시키는 정보 탐색·질문 병목에 집중**한다.
즉, 위드버디의 목표는 조기퇴사의 모든 원인을 제거하는 것이 아니라, 모르는 것이 생겼을 때 즉시 물어볼 수 없는 상황을 줄이는 것이다. 이 병목은 기술로 개입 가능하고, 해결됐을 때 신입의 심리적 장벽과 사수의 반복 응대 부담을 동시에 낮출 수 있다.

---

### 💼 기업이 겪는 구조적 문제

위드버디의 실제 구매자는 기업이다. 신입사원이 서비스를 사용하지만, 도입 결정과 비용 지불은 기업이 한다.

**기업의 세 가지 Pain Point**

1. **사수의 반복 응대는 비용으로 작동한다**

사수의 반복 응대는 단순한 시간 소모가 아니라 **업무 흐름을 끊는 비용**이다. 자체 설문에서 사수의 **67.4%**가 신입 질문 응대로 업무 영향을 받는다고 답했고, 동일 기초 정보 반복 안내 경험 4점 이상(5점 만점)은 **62.6%**이었다. 이 중 16.0%는 "업무 흐름이 자주 끊겼다"고 응답했는데, 한 번 끊긴 집중을 회복하는 비용까지 고려하면 실제 손실은 더 크다. 작은 기업일수록 직원이 멀티롤을 맡기 때문에 집중 업무가 끊기는 비용이 더 크다. 이 반복 응대의 대부분은 회사에 대한 정책·절차·세팅형 질문으로, 답이 명확하고 자동화 적합도가 높다.

2. **문제의 형태는 회사마다 다르지만 사수 의존이 발생하는 결과는 같다**

사수에게 "신입이 가이드 문서가 있어도 사수에게 직접 묻는 이유"를 물었을 때 가장 결정적인 발견이 나왔다. 그리고 이 발견은 **회사 규모에 따라 다른 양상**을 보인다. 위드버디는 이 두 시장에 단계적으로 접근한다. 문서가 있는 기업에서는 **탐색 실패**(57.2%)가 사수 의존을 만든다. **이 시장에서는 문서를 더 만드는 것보다 자연어 질문으로 답을 꺼내주는 도구가 우선 필요하다.** MVP의 RAG Q&A가 이 문제를 푼다. 문서가 부족한 기업에서는 **문서 부재**가 사수 의존을 만든다. 사수가 답할 뿐 아니라 답해야 할 내용을 직접 만들어야 한다. 이 시장은 RAG만으로 해결되지 않으며, 문서 생성과 구조화를 지원하는 별도 접근이 필요하다. 고도화 단계의 미응답 분석·템플릿 제공·문서화 지원이 이 문제를 푼다. 두 시장 모두 결과적으로 사수에게 부담이 집중되는 구조이며, 위드버디는 단계적으로 두 시장에 접근한다.

3. **사수가 시간을 쓰는 영역의 절반 이상은 자동화 가능한 영역이다**

사수가 신입 온보딩에 가장 많은 시간을 쓰는 영역을 묻는 질문에서, 실무 코칭(86.6%)을 제외한 나머지 영역 중에서 자동화 가능한 영역(IT 환경 설정 + 회사 제도 안내)은 사수의 시간을 빼앗는 주요 영역으로, 응답률을 합치면 56.2%다. **사수의 시간 부담 절반은 위드버디로 회수 가능한 영역**이라는 의미다.

## 3. 리서치 (Research)

### 3.1 데스크리서치

**① 초기 적응 실패는 반복되는 문제다 — 조기 퇴사는 그 결과 중 하나다.**

신입의 조기 이탈은 규모를 불문하고 반복적으로 관찰되는 현상이다. 국내 기업 신입직 조기퇴사율은 약 28.87%로 보고된 바 있으며(KRIVET 계열 연구, 관측 기준 2020년대 초), 대졸 신입 기준 300인 미만 기업의 퇴사율(32.5%)은 300인 이상 기업(9.4%)의 3배 이상이다(한국경영자총협회, 2016; 이후 복수 조사에서 유사 추세 확인).

다만 조기퇴사의 원인은 단일하지 않다. 복수 조사에서 1위는 일관되게 직무·조직 적응 실패로 나타나며, 여기에는 직무 적합성 불일치, 조직 문화, 낮은 보상, 성장 기대 차이 등이 함께 포함된다. 위드버디는 그중에서도 기술로 비교적 빠르게 개입 가능한 병목인 **정보 접근 실패와 질문 창구 부재에 집중**한다.

온보딩 개입이 이탈에 영향을 줄 수 있다는 근거는 존재한다. Cable et al.(2013)의 Wipro BPO 실험(n=605)에서는 입사 초기에 심리적으로 지지받는 환경을 경험한 집단이 표준 온보딩 집단 대비 6개월 내 이탈률이 더 낮았다. 또한 강력한 온보딩 프로세스가 신규 입사자 유지율 향상 및 생산성 개선과 연관된다는 결과는 Brandon Hall Group 계열 연구에서 반복적으로 인용된다. 다만 위드버디의 효과는 외부 연구를 그대로 전이하기보다, MVP 실측 데이터로 별도 검증하는 것을 원칙으로 한다.

**② 창구 부재는 신입과 사수 양쪽의 손실로 이어진다**

신입 업무 적응 어려움 원인 1위는 '가르쳐주는 사람이 없어서'(49.3%), 2위는 **'조언 구할 선배사원이 없어서'(32.4%)** 였다(사람인, 2022). 그런데 이 문제는 성격이 서로 다른 두 층위로 나뉜다.

| 층위 | 내용 | 해결 방식 |
| --- | --- | --- |
| Layer 1 — 업무를 모른다 | 직무 실무, 의사결정 방식, 조직 맥락 | 사람의 맥락 공유와 대면 교육이 더 중요한 영역 |
| Layer 2 — 기반 정보를 모른다 | 연차 규정, 비품 신청, IT 설정 등 정책·절차·세팅형 정보 | 반복성과 정답 소스가 있어 초기 자동화에 상대적으로 적합 → 위드버디의 우선 집중 영역 |

사수 관점에서도 반복 응대 부담을 가장 먼저 줄일 수 있는 영역은 Layer 2다. 자체 설문에서 사수의 **67.4%** 가 신입 질문 응대가 업무에 영향을 준다고 응답했고, 반복 안내 빈도는 평균 **3.70점(5점 만점)** 이었다. 업무가 한 번 중단되면 다시 흐름을 회복하는 데 비용이 발생한다는 점을 고려하면, Layer 2 질문은 반복성이 높고 정답 소스가 존재하기 때문에 운영 효율 개선을 가장 먼저 측정하기 좋은 자동화 대상이다.

**③ 왜 하필 이 질문군인가 : 답변 가능성으로 선택했다**

위드버디 v.1의 처리 범위는 문서 기반으로 답변 가능하고 반복성이 높은 질문군이다. 초기에는 인사·행정·복리후생·IT 문서에서 발생하는 정책·절차·세팅형 질문을 우선 공략하고, 이후 표준작업형, 판단 보조형, 코칭형 질문으로 확장한다.

| 선택 기준 | 설명 |
| --- | --- |
| 반복성이 높다 | 신입마다 유사 질문이 반복적으로 발생한다 |
| 공식 근거가 존재한다 | 회사 내부 문서, 규정집, 가이드 등 답변 소스가 있다 |
| 문서화·구조화가 가능하다 | RAG가 처리하기 적합한 형태로 정리할 수 있다 |
| 오답 리스크를 통제할 수 있다 | 정답 범위가 비교적 명확해 신뢰도를 높이기 쉽다 |
| 운영 ROI를 측정하기 쉽다 | 반복 응대 감소 효과를 볼 수 있다 |

인사·행정·복리후생·IT는 이 조건을 만족하는 질문이 주로 존재하는 문서 소스 도메인이다. 즉 범위 선택의 기준은 부서가 아니라 답변 가능성이다. 이는 심층인터뷰를 통해서도 확인 할 수 있었다.

> "SOP(**표준운영절차)**가 있었지만 어떤 내용을 찾기 위해 무슨 SOP를 봐야 하는지를 몰라서 사수한테 질문하는 경우가 되게 많았어." — 제약회사 QA 7년차
> 

즉, 문제는 단순히 문서의 유무가 아니다. 문서가 있어도 탐색이 안 되면 결국 사람에게 묻게 된다. 위드버디의 RAG가 필요한 이유는 문서를 새로 만들기 위해서가 아니라, 흩어진 공식 정보를 신입의 자연어 질문 기준으로 다시 꺼내 주기 위해서다.

**④ 시장·경쟁 시그널 — 이 문제는 성립하지만, 위드버디의 포지셔닝은 다르다**

시장에는 이미 사내 문서 기반 질의응답을 자동화하는 범용 업무 AI가 등장하고 있다. 이는 규정·절차·반복 문의 자동화 자체는 유효한 사용 시나리오임을 보여준다. 다만 범용 업무 비서는 전사 임직원을 대상으로 폭넓은 업무 생산성을 다루는 반면, **위드버디는 입사 초기 30~90일의 심리적 맥락과 질문 장벽에 특화된 온보딩 에이전트**다.

위드버디의 차별화 포인트는 단순 문서 검색이 아니라 다음 세 가지에 있다.

- **Buddy Nudge**: 질문 전 단계에서 먼저 말을 걸어 진입 장벽을 낮춘다
- **온보딩 시점 특화**: 입사일 기준 D+0, D+3, D+7, D+30 맥락을 반영한다
- **사수 연결 브리지**: AI가 답하기 어려운 것은 구조화해서 사람에게 넘긴다

즉, 범용 사내 AI가 '전사형 업무 비서'라면, 위드버디는 **'신입의 첫 질문 창구'**를 만드는 제품이다.

### 3.2 사용자 조사 (설문조사 n=277 / 심층인터뷰 n=9)

**신입 응답 주요 결과 (n=90)**

1. 입사 초기 가장 큰 어려움 1위: 눈치 보여서 묻기 어려움 (41.1%)
2. 혼자 해결하려다 시간 낭비: 27.8%
3. 질문 해결 방법 1위: 사수/리더에게 직접 질문 (50%)
4. AI 에이전트 도움 기대 4점 이상: 83.3%

**사수 응답 주요 결과 (3년 차 이상의 신입 교육·응대 경험이 있는 구성원, n=187)**

1. 반복 안내 빈도 4점 이상: 62.6% / 평균 3.70점(5점 만점)
2. 질문 응대가 업무에 영향을 준다: 67.4% (일정에 어느 정도 영향을 줬다 51.3%, 업무 흐름이 자주 끊겼다 16.0%)
3. 사수가 본 신입이 사수에게 묻는 이유: “탐색 실패와 직결된 응답”(57.2%), “사수에게 묻는 게 가장 빠르기 때문"(33.7%) - 거의 모든 사수 의존이 정보 접근 효율의 문제로 귀결
4. AI 대체 시 생산성 도움 기대 4점 이상: 77%

**리서치 한계**

본 리서치는 통계적 증명보다 방향성 탐색을 목적으로 한다. 설문 모수(n=277)는 전체 모집단 대비 제한적이며 대표성을 보장하기 어렵다. 또한 본 리서치는 문제의 존재와 행동 패턴은 보여주지만, 정책·절차·세팅형 질문이 실제로 어느 정도 비중과 반복성을 갖는지까지는 직접 측정하지 못했다.

따라서 다음 단계에서는 파일럿 기업의 실제 질문 로그를 수집해 질문 유형별 비중, 반복률, 문서 기반 답변 가능률, 사람 개입 절감 효과를 실측해야 한다. 또한 '질문 창구 부재가 조기퇴사로 이어진다'는 인과관계는 MVP 이후 장기 지표로 검증하며, 현재 PRD에서는 직접 효과가 아니라 초기 적응 부담 완화와 운영 효율 개선 수준으로 주장 높이를 제한한다.

## 6. 솔루션 개요 (Solution Overview)

### 6.1 서비스 설계 원칙

**1) v1의 범위는 가장 빠르게 신뢰를 만들 수 있는 영역부터 시작한다**

위드버디는 신입이 눈치 보며 삼켰던 질문에 즉시, 정확하게 답할 수 있는 영역부터 시작한다. 정책·절차·세팅형 질문은 공식 근거가 존재하고 반복성이 높아 AI가 가장 먼저 신뢰를 만들 수 있는 영역이다. 이 첫 경험이 쌓이면 신입은 위드버디를 안전한 첫 질문 창구로 인식하게 된다.

**2) 핵심 ICP는 문서는 있지만 전달할 인력이 부족한 조직이다**

위드버디 MVP의 핵심 타깃은 규정집·가이드·공지 등 사내 문서는 존재하지만, 신입에게 일일이 설명해줄 전담 인력(HR팀, 경영지원팀)이 없는 30~100인 규모 조직이다. 문서가 있어도 신입이 어디서 무엇을 찾아야 할지 모르고, 결국 사수가 반복 설명해야 하는 구조가 위드버디가 가장 먼저 해결하는 지점이다.
문서가 거의 없는 조직은 MVP 범위 밖이며 이 세그먼트는 고도화 단계에서 표준 온보딩 템플릿 제공과 관리자 문서 데이터화 지원으로 별도 접근한다.

| 단계 | 설계 | 내용 | 대상 |
| --- | --- | --- | --- |
| MVP | 회사 문서 기반 RAG+근로기준법 기반 범용 답변 레이어 | 업로드된 규정·가이드·공지를 기반으로 신입의 자연어 질문에 즉시 답변. 회사 규정을 1순위로 노출하고 법령과의 정합성을 함께 확인 | 문서는 있지만 전달할 인력이 없는 조직  |
| 고도화 ① | 표준 온보딩 템플릿 제공 | '우리 회사 연차는 ○일입니다' 형태의 빈칸 채우기 템플릿 제공. 관리자가 최소한의 입력만으로 RAG를 작동시킬 수 있게 함 | 문서가 거의 없는 소규모 조직 |
| 고도화 ② | 관리자 문서 데이터화 지원 | 미답변 로그 분석 기반으로 자주 묻는 항목을 파악하고 필수 문서 구조화를 지원 | 문서가 흩어져 있거나 정리가 안 된 조직 |

**3) 아하 모먼트는 '첫 번째 정확한 답변 + 안심감'에 있다**

신입이 사수에게 물어보기 눈치 보였던 것을 위드버디에게 물었을 때, 회사 규정 기반의 정확한 답변이 즉시 나오고 동시에 안전하게 질문해도 된다는 감각을 얻는 순간이 아하 모먼트다. 단, 회사 문서가 등록되지 않은 경우 MVP 단계에서는 근로기준법 기반 범용 답변만 제공되며, 회사 특화 답변 기반의 완전한 아하 모먼트는 최소한의 문서 등록 이후 가능하다.

**4) 위드버디는 신입 교육 도구가 아니라 기업의 반복 응대 자동화 도구다**

위드버디의 도입 근거는 "사수가 같은 질문에 답하는 시간이 줄어든다"에 있다. 이 포지셔닝이 작은 기업의 도입 장벽을 낮춘다. 기업 입장에서 "교육 도구"는 새로운 부담(누가 운영할 것인가)을 만드는 반면, "자동화 도구"는 기존 부담(사수의 반복 응대)을 줄이는 것이며 위드버디는 후자를 명확히 한다.

### **7.2 측정 근거 및 설명**

1. **MVP 핵심 집중: P0**
- **My Buddy - RAG Q&A:** RAG Q&A RICE 점수(41.7)는 Buddy Nudge(62.5)보다 낮지만, RAG Q&A는 위드버디의 핵심 해결 가치다. Buddy Nudge가 질문을 시작하게 만드는 UX 레이어라면, 실제로 “회사 규정 기반의 정확한 답변”을 제공해 문제를 해결하는 서비스 엔진은 RAG다. Quick Tap도 최종적으로는 RAG 답변으로 연결되므로, 구현 난이도가 높더라도 P0로 둔다.
- **My Buddy - Buddy Nudge :** Buddy Nudge는 신입의 첫 질문 진입 장벽을 낮추는 핵심 관문이다. D+0/D+3/D+7/D+30 맥락에 맞는 선제 안내와 Quick Tap 버튼을 통해 “먼저 물어봐도 되는 분위기”를 만들고, RAG 경험으로 자연스럽게 연결한다. Rule-based로 구현 가능해 MVP에서 안정적으로 검증하기 적합하다.
- **로그인 :** 로그인은 직접적인 핵심 기능이라기보다, Buddy Nudge와 이후 데이터 축적을 가능하게 하는 기반 기능이다. 입사일 정보가 있어야 Day 기반 선제 안내가 가능하고, 사용자 식별이 있어야 질문 로그를 축적해 이후 AI Buddy Nudge 고도화로 이어질 수 있다.
- **간소 관리자 페이지 :** 기업 관리자가 신입 계정을 생성하고 고도화 때 회사 문서를 업로드할 수 있는 최소 기능 페이지. 관리자가 회사코드, 신입 이름, 사원번호, 입사일을 입력하면 계정이 생성된다. 생성된 계정 정보를 신입에게 직접 전달하면 로그인 가능하다.
1.  **이번 MVP에선 제외: P1**
- **관리자 문서 데이터화(문서 파편화 또는 없는 문서) :** 어떤 문서를 어떤 구조로 넣어야 RAG가 잘 작동하는지는 운영 전에는 완전히 정의하기 어렵다. MVP에서는 최소한의 문서 업로드만으로도 핵심 가설을 검증할 수 있으므로, 미답변 로그가 쌓인 뒤 템플릿 UI와 필수 문서 구조화를 고도화 항목으로 둔다.
- **AI Buddy Nudge :** 개인화 Nudge는 질문 로그와 사용 패턴이 충분히 쌓여야 학습 가치가 생긴다. 따라서 MVP에서는 Rule-based로 먼저 검증하고, 이후 데이터 축적을 전제로 AI 전환을 검토한다.
- **질문 코칭 :** 질문 코칭은 회사별 암묵지와 소통 관례가 반영되어야 가치가 커지며, 현재는 구현 방식도 확정되지 않았다. 따라서 MVP에서는 직접 답변 가능한 질문군에 집중하고, 질문 코칭은 확장 단계의 차별화 기능으로 둔다.

v1의 처리 범위는 인사·행정·복리후생·IT 등 이 질문군이 주로 존재하는 문서 소스 도메인이다.

| **구분** | **카테고리** | **예시 질문** |
| --- | --- | --- |
| **IN SCOPE** | **인사 (HR)** | 연차는 언제부터? / 수습 기간은? / 수습 평가 기준은 |
| **IN SCOPE** | **행정** | 비품 신청은? / 경비 처리는? / 결재 라인은? |
| **IN SCOPE** | **복리후생** | 식대 지원은? / 건강검진은? / 야근 수당은? |
| **IN SCOPE** | **IT 환경 및 예약** | 이메일 계정은? / 회의실 예약은? / VPN 설정은? |
| OUT OF SCOPE | 직무 실무 | 회사마다 맥락 달라 표준화 불가.오답이 나올 가능성 높음 |
| OUT OF SCOPE | 법률·세무 자문 | 정보 제공 아닌 상담 영역전문가 자문 필요. 오답 시 법적 책임 문제 |
| OUT OF SCOPE | 대인관계, 개인 상담 | 암묵지 비중 높음 |

### 8.1.1 RAG Q&A — AI 구현 (P0 최우선)

회사 관리자가 업로드한 문서를 RAG 처리하여 신입의 자연어 질문에 즉시 답변한다. 입사 초 필요한 IN SCOPE 카테고리 내 질문만 처리하며 주요 문서 소스 도메인은 인사·행정·복리후생·IT다. 관련 양식이 DB에 있으면 답변 하단에 자동 제공한다.

**답변 원칙**

1. 회사 문서 우선: 회사 문서가 있으면 해당 문서를 우선 근거로 사용한다
2. 공통 문서 보완: 회사 문서가 없으면 공통 문서 또는 최저임금법, 근로기준법, 기간제법, 남녀고용평등법, 퇴직급여법 기반 범용 답변으로 보완한다
3. 범위 밖 질문 거절이 아니라 전환: 판단형·관계형·실무 코칭형 질문은 답변하지 않고, 사람에게 물어볼 수 있도록 구조화해 안내한다

**질문 유형별 기대 응답**

| 항목 | 내용 |
| --- | --- |
| IN SCOPE 처리 |   • 정책형·절차형·세팅형 질문(인사, 행정, 복지, it)을 대상으로 한다. 회사 문서를 1순위 근거로 사용하고, 없으면 공통 문서로 보완한다. 답변 메시지는 `messageType = rag_answer`로 저장되며, 근거 문서는 `documentIds`와 `documents[]`로 함께 반환한다. 
  • 회사 문서 우선 답변 + 출처 표시 |
| OUT OF SCOPE 처리 |   • 판단형·관계형·실무 코칭형 질문은 답변을 거절하지 않고, 사람에게 물어볼 수 있도록 구조화해 안내한다. `messageType = out_of_scope`로 저장 |
| 문서 부족 처리 |   • 질문 범위는 IN SCOPE이지만 근거 문서나 정보가 없는 경우, 근로기준법 기반 범용 답변으로 보완하고 회사 규정 확인을 안내한다. `messageType = no_result`로 저장 |
| 근거 문서 연결 |   • 답변 메시지와 근거 문서는 `chat_message_documents`를 통해 N:N으로 연결된다. `documentIds`는 문서 ID 목록, `documents[]`는 상세 정보를 담는다 |
| 아하 모먼트 |   • 회사 문서 기반의 정확한 답변이 즉시 나오는 첫 순간. 동일 의도 재질문 없이 세션이 종료되는 비율로 간접 측정 |

**인수 조건 (Acceptance Criteria)**

1. 자연어 질문 입력 후 5초 이내 답변 생성 
2. IN SCOPE 50개 샘플 테스트 정답률 80% 이상
3. OUT OF SCOPE 질문 → 안내 메시지 노출 정확도 70% 이상
4. 관련 양식 DB에 있으면 답변 하단 자동 노출 + 다운로드
5. `rag_answer` 하단에 '도움이 됐어요 / 더 확인이 필요해요' 버튼 노출 및 클릭 로그 기록

### 8.1.2 Buddy Nudge

MVP에서는 AI 없이 Rule-based 조건 로직으로 구현. 입사일 조건에 따라 미리 설계된 메시지와 버튼을 출력한다.

| 입사일에 따른 트리거 | 출력 내용 예시 |
| --- | --- |
| D+0 최초 로그인 | Welcome 메시지 + '입사 첫날 많이 묻는 것들이에요' 카드 + Quick Tap 버튼 3개 (IT 세팅·비품·연차) |
| D+3 로그인 | '3일차에 많이 생기는 궁금증이에요' 카드 + 행정·복지 중심 버튼 |
| D+7 로그인 | '첫 주가 지났어요' 카드 + 버튼 |
| D+30 로그인 | '한 달이 됐네요' 카드 + 버튼 |
| 상시 | Quick Tap 버튼 노출 → 클릭 시 질문 자동 채워짐 → RAG 즉시 연결 |

**인수 조건 (Acceptance Criteria)**

- 최초 로그인 시 Welcome 메시지 노출
- Day 조건별 카드 정상 출력 (D+0, D+3, D+7, D+30)
- Quick Tap 클릭 → 질문 자동 채워짐 → RAG 연결
- Day 카드 노출 시 `chat_messages`에 `message_type = suggestion`으로 저장
- Quick Tap 버튼 클릭 시 `user_activity_logs`에 `BUTTON_CLICK` 이벤트 기록

### 8.2 로그인

두 가지 목적: ① 입사일 정보로 Buddy Nudge Day별 트리거 ② 질문 데이터 DB 축적 → 고도화 AI Buddy Nudge 학습 기반

**인수 조건 (Acceptance Criteria)**

- 신입 계정 생성: 회사코드, 이름, 사원번호 (MVP에서는 백엔드가 직접 계정 생성)
- 관리자 UI를 통한 이메일 초대 및 계정 생성은 고도화로 이동
- 로그인 후 입사일 기준 Day 카운트 자동 계산 → Buddy Nudge 트리거
- 로그인 성공 시 `user_activity_logs`에 `event_type = SESSION_START`, `session_type = LOGIN` 기록
- 채팅 화면 진입 시 `POST /api/v1/chat/session-start` 호출 → `event_type = SESSION_START`, `session_type = CHAT` 기록 (30분 이내 재진입 중복 제외)
- 로그아웃 API 및 Refresh Token은 고도화 항목. MVP에서는 토큰 만료 시 로그인 페이지로 리다이렉트 처리.

### 8.3 간소 관리자 페이지 - 신입 계정 생성

관리자가 회사코드, 신입 이름, 사원번호, 입사일을 입력하면 계정이 생성된다. 생성된 계정 정보를 신입에게 직접 전달하면 로그인 가능하다. MVP에서는 신입 계정 생성 UI만 구현한다. 문서 업로드 기능은 백엔드 API가 완성된 상태이며, UI는 고도화에서 관리자 대시보드와 함께 구현한다.

**입력 필드**

- 회사 코드 (자동 채워짐, 변경 불가)
- 신입 이름
- 사원번호
- 입사일

**인수 조건 (Acceptance Criteria)**

1. 관리자가 5분 이내에 신입 계정 1개를 등록할 수 있다
2. 등록된 계정으로 신입이 즉시 로그인 가능하다
3. 입사일 기준 D+0 카운트가 자동 시작된다
4. 동일 회사 내 사원번호 중복 시 에러 메시지 노출

### 8.4 사용자 라이프사이클

위드버디는 신입사원이 입사 후 수습기간인 **일정 기간 동안만** 사용하는 도구다. 무기한 사용을 전제하지 않으며, 다음과 같은 라이프사이클을 가진다.

### 8.4.1 단계별 정의

| 단계 | 기간 | 위드버디 활용 | 정상 신호 |
| --- | --- | --- | --- |
| **학습기** | D+0~7 | 적극 사용 (다수 질문) | 활발한 인터랙션 |
| **적응기** | D+8~30 | 점진 감소 | 질문 수 감소 추이 |
| **자립기** | D+30~수습 종료 | 간헐적 사용 | 월 1회 이상 방문 |
| **졸업기** | 수습 종료 + 30일 후 | 비활성화 | 읽기 전용 모드 |

### 8.4.2 자동 비활성화 정책

수습 종료 후 30일 버퍼를 거쳐 자동 비활성화된다.

**비활성화 방식: 읽기 전용 모드**

- 이전 대화 기록은 볼 수 있음
- 새 질문 입력 불가
- 안내 메시지 노출: "수습 기간이 종료되어 새로운 질문은 받을 수 없어요. 추가 도움이 필요하면 사수나 HR에 문의하세요."

**비활성화가 필요한 이유**

1. **위드버디의 정체성 유지**: 신입 전용 도구로서의 명확성
2. **회사별 신입 풀 기반 가치**: 한 명이 졸업하면 다음 신입이 가치를 이어받음
3. **데이터 노이즈 제거**: 비활성 계정의 누적 방지

### 8.4.3 회사별 수습기간 설정

회사마다 수습기간이 다르므로 관리자가 설정 가능하다.

- 우리 회사 수습기간: 1개월 / 3개월 / 직접 입력
- 비활성화 버퍼: 기본 30일
- 비활성화 후: 읽기 전용 / 완전 차단

기본값은 **수습기간 3개월 + 30일 버퍼 = D+120 비활성화**로 설정한다.

| TC-03 | 절차형 질문 | '비품 신청은 어떻게 하나요?' | 답변 + 신청서 양식 자동 노출 |
| TC-04 | 세팅형 질문 | '회사 이메일/인트라넷 계정은 어떻게 만드나요?' | 가입 방법 안내 |
| TC-05 | 판단형 질문 (OUT) | '이 개발 코드 어떻게 리뷰하나요?' | 안내 메시지 + 사람에게 물을 범위 제시 |
| TC-06 | Quick Tap 버튼 클릭 | D+0 카드 버튼 클릭 | 질문 자동 채워짐 → RAG 연결 |
| TC-07 | RAG 정확도 일괄 | IN SCOPE 50문항 + OUT OF SCOPE 20문항 | IN SCOPE 정답률 80% 이상 / OUT OF SCOPE 인식 정확도 70% 이상  |
| TC-08 | 관리자 신입 계정 생성 | 이름·사원번호·입사일 입력 후 생성 | 계정 생성 완료 / 해당 계정으로 즉시 로그인 가능 / 입사일 기준 D+0 카운트 자동 시작 |

### 10.2 엣지 케이스

| 케이스 | 기대 처리 |
| --- | --- |
| RAG 미답변 (DB에 정보 없음) | '확인이 필요한 내용이에요. 담당자께 여쭤보시거나, 관리자에게 추가 요청해 주세요' |
| 모호한 질문 ('연차 어떻게 해요?') | 즉시 제공 가능한 정책 정보를 먼저 주고 추가 선택 유도 |
| IN+OUT 복합 질문 | IN SCOPE 부분 답변 + OUT OF SCOPE는 안내 메시지로 분리 처리 |
| 관리자 문서 미등록 상태 | 근로기준법 기반 범용 답변 제공 + '우리 회사 규정은 관리자에게 확인해 주세요' |
| 문서 근거 상충 | 최신 문서 우선 적용 + 근거 문서 표시 + 확인 필요 안내 |
| 빈 입력 / 특수문자 전송 | 버튼 비활성화 / '질문을 입력해주세요' 안내 |
| RAG 타임아웃 (10초 초과) | '잠시 문제가 생겼어요. 다시 시도해주세요' + 재시도 버튼 |

### 11.2 비즈니스 모델

**위드버디의 가치는 "개별 신입이 평생 쓰는 도구"가 아니라 "회사가 신입을 받을 때마다 쓰는 인프라"**

회사는 매년, 매분기, 매월 신입을 받는다. 한 명이 자립하면 다음 신입이 들어오고, 또 다음 신입이 들어온다. 회사 단위로 보면 위드버디는 항상 활성 상태다. 그리고 누적된 질문 데이터가 다음 신입에게 더 정확한 답변을 제공하는 자산이 된다. 이 모델은 회사가 신입 채용을 지속하는 한 가치가 유지되며 개별 신입의 자립은 곧 다음 신입을 위한 데이터 자산 누적으로 이어진다.

| 구분 | 내용 |
| --- | --- |
| 수익 모델 | 기업(관리자) 구독 SaaS — 신입사원 수 or 기업 규모 기반 월 구독료 |
| 과금 단위 (예시) | 소기업(~30인): 월 X만 원 / 중소기업(~100인): 월 Y만 원 |
| 장기 구독 유인 | 질문 데이터가 기업 지식 자산으로 누적 →미해결 패턴 학습 → 해지 시 데이터 손실 → 자연스러운 Lock-in |
| MVP 목표 | 파일럿 2개사 이상 / 도입 의향 인터뷰 긍정 반응 확인 |

### 11.3 경쟁 포지셔닝

**범용 사내 AI vs 위드버디**

위드버디는 범용 사내 AI와 경쟁하려는 것이 아니라, 범용 사내 AI가 놓치기 쉬운 입사 초기 심리적 맥락과 질문 전 망설임에 특화된 온보딩 에이전트다.

| 항목 | 범용 사내 AI / 업무 비서 | 위드버디 |
| --- | --- | --- |
| 주요 사용자 | 전사 임직원 | 신입사원 + 직속 사수 |
| 사용 시점 | 상시 업무 생산성 | 입사 초기 30~90일 온보딩 |
| 사용 기간 | 상시 | 수습기간 한정 (수습 종료 후 자동 비활성화) |
| 핵심 가치 | 문서 검색, 업무 자동화, 생산성 향상 | 질문 장벽 완화, 첫 질문 창구, 초기 적응 지원 |
| AI 태도 | 유틸리티 중심 | Buddy 중심 |
| 차별화 UX | 사용자가 먼저 질문해야 작동 | Buddy Nudge로 먼저 말을 건다 |
| 범위 밖 처리 | 일반 안내 | 사수 연결 브리지 + 질문 코칭 |
